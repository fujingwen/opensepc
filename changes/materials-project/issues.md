# 工程项目问题记录

## 已确认问题

1. 字典迁移：`test.base_dictionarytype`（`F_IsTree=0`）及关联的 `test.base_dictionarydata` 需要迁移到 `master.sys_dict_type` 和 `master.sys_dict_data`，不能删除原始数据，编码冲突需单独标记。
2. 前端字典选择器仍需继续统一，尤其是“信息确认单位”当前存的是名称，不是标准字典编码，暂不宜直接字典化。
3. “有无填报”逻辑与真实库值仍有口径差异，`has_report` 需要继续统一到数据库实际字典值。
4. 一体化工程编码下拉已接入，但后端分页 SQL 仍按 `gc_code` 非空过滤，与设计要求按施工许可证口径不一致。

## 对照操作手册发现

以下问题为对照《青岛市建设工程材料信息管理平台操作手册》后新增确认：

1. 操作手册要求“施工单位”由当前登录施工单位自动带出，不允许人工从全量企业中重新选择；当前页面仍保留施工单位下拉选择，和手册流程不一致。

## 当前核查结论

1. 字典迁移相关能力应继续由独立提案 `dict-migration-test-to-master` 承接。
2. 工程项目页面与接口主体已切换到 `master.t_project`。
3. 一体化工程编码远程搜索、自动回填链路已具备，但仍需进一步统一筛选口径。

## 待验证事项

1. 权限控制验证：`materials:project:list`、`materials:project:query`、`materials:project:add` 等权限点。
2. “施工单位自动带出”改造后，对现有管理员/运维角色页面是否需要保留特殊维护入口。
3. 一体化工程编码历史数据中，`is_integrated` 与施工许可证匹配不一致的存量记录如何治理。

## 备注

1. 本文件中带“对照操作手册发现”的条目，表示问题来源于与操作手册逐项比对后的新增差异，不是单纯实现缺陷汇总。
