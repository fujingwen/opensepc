## Context

当前建材管理系统使用Vue 3 + Element Plus作为前端框架，Spring Boot作为后端框架，已有备案产品管理功能但缺少通用的批量导入能力。用户需要逐条录入数据，效率低下。系统已有用户导入功能作为参考（包括前端上传弹窗和后端Excel解析逻辑），可以复用现有的hny-common-excel模块和ExcelUtil工具类。

为了提高代码复用性，需要实现一个通用的批量导入组件，不仅支持备案产品导入，还能方便地扩展到其他数据类型的导入。

## Goals / Non-Goals

**Goals:**

- 实现三步式通用批量导入弹窗组件（前端）
- 实现通用批量导入后端框架
- 实现备案产品批量导入接口（作为第一个使用场景）
- 支持Excel文件（.xls/.xlsx）上传和解析
- 提供数据预览和编辑功能
- 支持下载导入模板
- 集成到备案产品管理页面
- 参考用户导入功能的实现方式
- 确保组件可在多个页面复用

**Non-Goals:**

- 不实现复杂的数据校验规则（基本格式校验即可）
- 不支持除Excel外的其他文件格式
- 不修改现有的用户导入功能

## Decisions

### 前端组件架构

- **决定**：创建独立的BatchImportDialog通用组件，使用Element Plus的el-dialog、el-steps、el-upload、el-table等组件
- **理由**：保持组件独立性，便于复用和维护；利用项目已有的Element Plus组件库

### 步骤设计

- **决定**：分为三个明确的步骤：上传文件、数据预览、导入数据
- **理由**：符合用户提供的UI设计，流程清晰，用户体验良好

### 数据处理

- **决定**：前端负责文件上传和数据展示，后端负责Excel解析和数据持久化
- **理由**：前后端分离架构，后端已有Excel处理能力（hny-common-excel模块）

### 状态管理

- **决定**：使用Vue 3 Composition API + ref/reactive管理组件内部状态
- **理由**：与项目现有代码风格一致，轻量级，不需要引入额外状态管理

### 后端实现参考

- **决定**：完全参考SysUserController、SysUserImportVo、SysUserImportListener的实现方式
- **理由**：保持代码风格一致，复用已有的ExcelUtil工具类和导入模式

### 后端架构

- **决定**：先创建通用导入框架（GenericImportVo、GenericImportListener），再基于框架实现备案产品导入
- **理由**：提高代码复用性，方便后续扩展到其他数据类型的导入

### 通用框架设计

- **决定**：创建通用导入VO基类和监听器基类，封装通用的导入逻辑
- **理由**：减少重复代码，提高开发效率

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| Excel文件解析失败 | 提供清晰的错误提示，支持重新上传 |
| 大数据量导入性能问题 | 限制单次导入数量（如1000条），显示导入进度 |
| 数据校验不完整 | 提供模板下载，明确告知用户数据格式要求 |
| 前后端联调问题 | 参考用户导入的接口设计，保持API一致性 |
| 通用性与特殊性平衡问题 | 通用框架提供基础功能，具体实现可通过继承和重写扩展 |
