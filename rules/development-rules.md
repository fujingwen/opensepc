# 开发落地规范

## 核心开发原则

- **严格遵循设计文档**：所有在系统（前端、后端、数据库等各环节）内部落地的具体代码都必须严格照搬并在逻辑上履行设计文档中定型的条例文档标准。
- 如果没有获得允许确认与重规划，任何人或 AI **不得**随意添加、删除、或修改定型的出入参数字段、接口格式与查询条件。

## 开发流程与各阶段准则

### 1. 设计文档与构建预设

- 设计文档必须经过充分评估后方可介入开发，需涵盖：带完整数据类型及查询方式的条件表、显示格式规则及列表展示表、囊括表单校验或必需性的控制规范以及底层映射表等核心。
- 我们建议使用对应的自动化脚本及 AI 来生成并初始化项目结构。如果是人工书面开发或更新操作，依然得被绑定在此流程上。

### 2. 代码审查反馈阶段 (Code Review)

- AI 或人类开发者审查代码时必须左右参照着设计文档内容进行逐行验证（Field by Field 审查）。开发环节引入的各种未按设计的异常皆属于拦截事件（Blocker），必须要等它修复到一致之后才能进总支合并。
- *(注：原先配套的本地自动一致性校验工具已退役。现在以人工及大模型自动校对机制为检查主轴)*

### 3. 测试覆盖标准

- 测试必须囊括全部的接口 API 验证，所有的前端展示边界以及条件请求回馈测试；且测试抛出的结果，其数据颗粒度需契合设计的底层诉求预期。

## 代码具体规范细则

### 前端开发细项

- **查询与列表**：视图与列表数据的排列、展示文字以及查询参数的传递规则务必须要跟设计相呼应。字段对应的 Placeholder(预显示文字) 和 Label 名也要全复制。
- **表单控制约束**：表单在字段长度、验证条件体系架构下，必须与设计文档保持丝毫不差，包含对文件上传组件关于件上传受限及大小约束也须完整落实。
- **查询数据预处理机制传递规范**：
  - 发送至中台的请求必须封装并装载了所需的一应参数。
  - 对于带有起止范围的模糊查询要提前完成类型转置和扁平化分解，即：所有非时间格式数值范围字段传参为 `起始阶段_Min` / `结尾阶段_Max`；
  - 面向日期的检索区间必须分别分解挂载成 `Start` 与 `End` 的参数对送入查询后台。
- **字典系统串联**：在查询检索项内部所有关于多选/枚举类型的，必定接上既定的系统数据字典调用，杜绝临时魔法变量与自行造假。

### 后端开发细项

- **多租户业务层防御体系**：为了防范租户越界和安全脱逃，所有承载下发任务的 Entity （业务表实体）**强制都必须继承自** `TenantEntity` （系统内已构建继承依赖）；而在业务 SQL Mapper 动态绑定层内，必定要隐式补充上 `tenant_id` 的限制匹配机制。
- **API 通讯规范**：后置提供支持的 API 功能对应签名格式、访问请求方式、返还的数据对象等也务必对齐设计基准线开发。
- **底层数据库查询层封装控制**：
  - 为适配前端抛送的组合结构范围值字段（例如时间范围的 `XXX_Start / XXX_End`，以及金额/数字等层级的 `XXX_Min / XXX_Max` 属性），在实体的接受对象上，这些不属于数据库原生数据列的字段需打上注解标记限定脱敏：`@TableField(exist = false)`。
  - 动态范围过滤匹配的执行方向逻辑须严防出错：向右收口需运用 `>=` ，往左包抄采取 `<=` 进行安全控制。
  - 后部查询应对任何动态组装条件进行必要的 Null 判空前置查扣拦截。
  - 字典查询类目要求以 `=` 直接精准等效过滤；非主语类的常规查询项建议为防注入采取全字匹配构造体：`like concat('%', #{param}, '%')` 。所有数据库映射层的实体描述务必携带注解/注释。

## 常见错误排错检查

- 错误 1：多租户实现遗漏：Mapper XML 中丢掉了由全局截断的 `tenant_id` 查询限制或是实体实体缺失直接继承依赖关系。
- 错误 2：参数转换在前端没正确切成 `Min/Max` 等虚拟标识段或落后于处理；又或后端接口内忽略追加 `@TableField(exist=false)`，两头未做同频导致接口映射失败发生抛锚或崩溃。
- 错误 3：查询条件和底层设计文件擅自变动，越界未通报就盲目进展开工上线。
