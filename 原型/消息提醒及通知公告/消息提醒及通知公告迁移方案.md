# 消息提醒及通知公告迁移方案

## 1. 模块范围

本模块对应迁移清单中的以下功能：

- 消息提醒及通知公告-审核消息提醒
- 超48小时未审核消息提醒
- 消息提醒及通知公告-通知
- 消息提醒及通知公告-系统公告
- 预警信息

## 2. 当前现状

### 2.1 已有能力

当前新系统已经有“通知公告”基础能力：

- 后端：`SysNoticeController`
- 前端：`views/system/notice/index.vue`
- 菜单：`system/notice`
- 数据表：`master.sys_notice`
- 当前记录数：`6`

这说明“系统公告”可以优先复用 `sys_notice`，不需要另起一套公告表。

### 2.2 旧系统仍在使用的表

旧系统消息相关表还保留在 `test`：

- `test.t_message`：17
- `test.base_message`：2
- `test.base_messagereceive`：177111

结论：消息中心和预警能力还没有迁入 `master`，只是旧库有数据。

### 2.3 当前缺失的能力

代码中没有发现以下正式模块：

- 消息中心页面
- 审核消息提醒页面
- 超时未审核提醒规则
- 预警信息页面
- 与建材审核联动的消息发送服务

虽然工作流模块里有 WebSocket、邮件、短信的通用消息能力，但还没有与建材业务模块打通。

## 3. 迁移判断

### 3.1 系统公告

- 现状：已承载
- 迁移建议：旧系统公告类能力优先映射到 `master.sys_notice`
- 注意：需要补角色维度，确认“建管中心公告 / 采购信息公告 / 库存信息公告”是否都能被 `sys_notice.notice_type` 或扩展字段覆盖

### 3.2 通知与消息中心

- 现状：未承载
- 迁移建议：不能直接宣称“新系统已具备”
- 需要先决定承载策略：
  - 方案 A：在 `master` 保留 `t_message`、`base_message`、`base_messagereceive` 迁移旧数据
  - 方案 B：重构为统一消息中心表，再做旧表数据映射

当前阶段更建议先采用方案 A，先保业务连续，再考虑归并。

### 3.3 预警信息

- 现状：未承载
- 数据来源倾向：旧系统 `base_message` 的 `F_Type=3`
- 迁移建议：先在 `master` 落旧表或统一消息表，再补独立预警列表页

### 3.4 审核消息提醒与超48小时提醒

- 现状：未实现
- 原因：
  - 现有审核接口还是占位
  - 没有超时规则任务
  - 没有面向建材审核的消息投递服务

结论：这两项必须排在“审核流落地”之后。

## 4. 推荐落地顺序

### 4.1 第一步：先分清公告与消息

- 公告：先映射到 `sys_notice`
- 消息/预警：单独保留旧业务语义，不要强行塞进 `sys_notice`

### 4.2 第二步：补 `master` 侧消息承载

建议在 `master` 补齐以下内容之一：

- 保守方案：直接迁 `t_message`、`base_message`、`base_messagereceive`
- 统一方案：建设新的消息中心表，再做旧表映射和兼容

### 4.3 第三步：与审核流联动

在建材填报审核真实落地后，再补下面两类能力：

- 审核状态变化消息提醒
- 超48小时未审核提醒

### 4.4 第四步：补页面

建议新增或恢复以下页面：

- 消息中心
- 预警信息
- 审核提醒列表
