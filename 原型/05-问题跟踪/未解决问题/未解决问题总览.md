## 2. 主流程总览

当前系统主流程仍可概括为：

`登录系统 -> 完善企业信息 -> 生产单位维护代理商 / 施工单位维护工程项目 -> 施工单位填报建材产品 -> 代理商/生产单位审核 -> 消息提醒 -> 公告查看/信息发布 -> 统计分析`

围绕这条主链路，当前仍未收口的问题主要集中在：

1. 企业和角色上下文还没有完全按手册收敛。
2. 工程项目与建材产品的若干关键业务口径仍未统一。
3. 消息提醒、公告发布与角色边界仍和原型流程有差异。
4. 若干基础能力和迁移类 change 文档状态已落后于代码、数据库现实。

## 3. 按业务流程归并的问题

### 3.1 登录后完善企业信息

涉及 change：

1. `base-companyinfo`
2. `system-region`
3. `user-role-migration`

未解决问题：

1. 企业信息模块当前仍是“后台主数据维护模式”，实际支持维护施工企业、生产企业、代理商三类全量主数据；这比原型中的“企业完善自身信息”范围更大。
2. 代理商维护仍允许从全量生产企业中选择所属企业，没有绑定到“当前生产单位上下文”。
3. 历史 `area` 脏数据映射到 `base_province` 标准地区名称的兜底仍不足。
4. 地区模块的搜索语义仍未优化：
   - `parentId` 为空时仍默认只查 `parent_id='-1'`
   - `en_code` 仍按精确匹配
   - 历史 `parent_id is null` 根节点数据仍未治理
5. `RegionSelect` 与原型中的 `common_region_picker` 仍存在交互差异，尚未统一。
6. 用户角色迁移文档状态与数据库现实不一致：proposal 仍偏“待执行”，但数据库已呈现“部分完成但未闭环”。
7. 用户迁移闭环未完成：
   - `test.base_user = 5233`
   - `master.sys_user = 4940`
   - `master.sys_user.original_id is not null = 4915`
   - rollback 验证仍未完成

### 3.2 生产单位维护代理商

涉及 change：

1. `base-companyinfo`

未解决问题：

1. 原型要求生产单位创建和维护“本单位代理商”，当前实现仍更接近后台统一管理模式。
2. 代理商与生产单位的业务归属仍未完全固化到“当前登录主体”。

### 3.3 施工单位维护工程项目

涉及 change：

1. `materials-project`
2. `dict-migration-test-to-master`

未解决问题：

1. “施工单位”仍保留全量企业下拉选择，未按手册改为从当前登录施工单位自动带出。
2. 一体化工程编码筛选口径仍未统一，后端分页 SQL 仍按 `gc_code` 非空过滤，与设计要求按施工许可证口径不一致。
3. “信息确认单位”当前保存的是名称，不是标准字典编码，尚未形成稳定字典化方案。
4. `has_report` 口径与数据库实际值仍存在偏差。
5. 历史 `construction_unit`、`is_integrated` 与施工许可证不一致的存量数据治理方案尚未明确。
6. `tasks.md` 仍停留旧口径，没有回写本轮审查结论。
7. 字典迁移配套仍未完全收口：
   - 新旧 `dict_type` 编码切换缺少完整验收闭环
   - 设计文档中的迁移结果统计与当前实库不一致
   - “允许迁移空字典类型”尚未写成明确验收约束

### 3.4 施工单位填报建材产品

涉及 change：

1. `materials-project-product`
2. `materials-record`
3. `batch-import-dialog`
4. `system-product`
5. `system-product-dict`

未解决问题：

1. 建材产品列表“重置信息确认状态”按钮及对应后端逻辑尚未实现。
2. 产品“规格 - 单位”数据模型仍未最终收口，当前基于 `master.sys_product.unit` 自动带出单位只是阶段性方案。
3. 供应商名称在部分场景应展示“无”的业务口径，仍未完成录入、保存、导出全链路核对。
4. `materials-project-product/tasks.md` 尚未回写，仍混有已经完成但未标记的旧任务。
5. 历史数据中仍有少量记录同时缺失 `batch_no` 和 `batch_date`，是否需要治理仍待确认。
6. 备案产品导入能力仍需与 `batch-import-dialog` 统一：
   - 模板下载
   - 上传
   - 预览
   - 编辑
   - 删除
   - 覆盖导入
   - 结果回传
7. `send_flag` 是否为正式业务字段仍待确认。
8. 企业删除口径尚未全链路统一到 `del_flag + company_type`。
9. 批量导入组件本身仍未完全收口：
   - 组件使用说明未补齐
   - 接入约束与复用边界未写清
   - 联调、边界测试结论未回写
   - 错误提示和逐行反馈体验仍可优化
10. 产品基础模型方向仍存在提案层冲突：

- `system-product` 已是代码与数据库中的真实运行模型
- `system-product-dict` 仍保留“产品能力仅依赖字典表”的旧方向
- 两个 change 尚未在文档层收敛为单一结论

11. `system-product` 自身仍存在文档口径分叉：

- `design.md` 仍写过独立 `/system/product/category`
- 文档仍出现过 `system/product/index1`
- 但当前代码、菜单 SQL、数据库菜单组件路径都收口为 `system/product/index`

12. `system-product` 使用的状态字典虽可运行，但文档没有固定字典选择，后续仍有再次分叉风险。

## 4. 优先级建议

### 第一优先级：直接影响主链路验收

1. `materials-project` 的施工单位上下文自动带出、工程编码筛选口径统一。
2. `materials-project-product` 的重置信息确认状态、规格单位模型、供应商名称口径收口。
3. `materials-record` 与 `batch-import-dialog` 的导入能力统一、`send_flag` / 删除口径收口。
4. `message-center` 的公告已读模型、历史迁移兼容、模板运行时支持。
5. `notice-announcement` 的后端企业上下文强约束、角色边界、真实数据验证。

### 第二优先级：影响提案可信度和后续验收

1. `quality-trace` 的 `specs/` 规范化和四页真实状态回写。
2. `user-role-migration` 的文档状态修正、差额核对、rollback 记录。
3. `dict-migration-test-to-master` 的验收口径回写。
4. `system-product` / `system-product-dict` 的方向统一。

### 第三优先级：工程化与体验优化

1. `batch-import-dialog` 的使用文档、边界测试、错误提示体验。
2. `system-region` 的搜索语义与组件统一。
3. `statistics-center` 的范围说明整理。
