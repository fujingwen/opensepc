## 3. 标签说明

### 3.1 风险等级

- `高`：直接影响主链路验收、真实业务结果或数据库数据正确性
- `中`：不阻断主链路，但会影响体验、提案可信度或后续验收
- `低`：偏工程化、文档或体验优化，短期不阻断主流程

### 3.2 责任归属

- `前端`
- `后端`
- `数据库`
- `文档`
- `运维`

单条问题可同时归属于多个责任面，写法统一为：

`[风险等级][责任归属] 问题描述`

例如：

`[高][后端/数据库] 数据闭环未形成，当前会直接影响查询和验收结果。`

## 4. 主流程与快速结论

当前系统主流程可概括为：

`登录系统 -> 完善企业信息 -> 生产单位维护代理商 / 施工单位维护工程项目 -> 施工单位填报建材产品 -> 代理商/生产单位审核 -> 消息提醒 -> 公告查看/信息发布 -> 统计分析`

围绕这条主链路，当前未收口问题主要集中在四类：

1. 企业、角色、地区等基础上下文还没有完全按操作手册收敛。
2. 工程项目、建材产品、备案产品之间的关键业务口径和数据库约束仍未完全统一。
3. 消息中心、通知公告、质量追溯等模块已形成主链路，但数据闭环和工程化仍不足。
4. 部分提案、任务文档和数据库现实已经脱节，仍需要继续回写收口。

## 5. 按业务流程归并的问题

### 5.1 登录后完善企业信息

相关 change：`base-companyinfo`、`system-region`、`user-role-migration`

未解决问题：

1. `[中][后端/文档]` 原型把企业信息模块定义成“企业用户维护本企业信息”，但当前实现更接近“后台统一维护全部企业主数据”，而且施工企业、生产企业、代理商三类都放在一个模块里管理，页面定位和权限边界与原型不一致。
2. `[高][后端]` 代理商维护仍允许从全量生产企业中选择所属企业，没有绑定到“当前生产单位上下文”。
3. `[中][后端/数据库]` 历史 `area` 脏数据映射到 `base_province` 标准地区名称的兜底仍不足。
4. `[中][后端/数据库]` 地区模块搜索语义仍未优化：`parentId` 为空时仍默认只查 `parent_id='-1'`；`en_code` 仍按精确匹配；历史 `parent_id is null` 根节点数据仍未治理。
5. `[低][前端]` `RegionSelect` 与原型中的 `common_region_picker` 仍存在交互差异，尚未统一。
6. `[中][文档]` 用户角色迁移相关文档还在按“准备开始迁移”来写，但数据库实际状态已经是“迁了一部分、还能用、但没有完全收尾”，文档和现实没有对齐。
7. `[高][数据库/文档]` 用户迁移还没有形成完整闭环：源库 `test.base_user = 5233`，开发库 `master.sys_user = 4937`，其中只有 `4915` 条带 `original_id`，`master.sys_user_role = 4938`，说明用户、原始 ID、角色关系还没有完整对上，rollback 也还没验证。

### 5.2 生产单位维护代理商

相关 change：`base-companyinfo`

未解决问题：

1. `[中][后端/文档]` 原型要求生产单位创建和维护“本单位代理商”，当前实现仍更接近后台统一管理模式。
2. `[高][后端]` 代理商与生产单位的业务归属仍未完全固化到“当前登录主体”。

### 5.3 施工单位维护工程项目

相关 change：`materials-project`、`dict-migration-test-to-master`

未解决问题：

1. `[高][前端/后端]` “施工单位”仍保留全量企业下拉选择，未按手册改为从当前登录施工单位自动带出。
2. `[高][后端]` 一体化工程编码列表现在还是“只显示 `gc_code` 不为空的数据”，但设计想要的是“按施工许可证口径来选项目”，两边筛选规则不是一回事。
3. `[中][前端/后端/文档]` “信息确认单位”目前存的是文字名称，不是标准字典值，后面做查询、回显、导出时都容易继续分叉。
4. `[中][后端/数据库]` `has_report` 口径与数据库实际值仍存在偏差。
5. `[中][数据库]` 历史 `construction_unit`、`is_integrated` 与施工许可证不一致的存量数据治理方案尚未明确。
6. `[低][文档]` `tasks.md` 仍停留旧口径，没有回写本轮审查结论。
7. `[中][数据库/文档]` 字典迁移的收尾还没做完：新旧 `dict_type` 编码切换没有完整验收记录，设计文档里的迁移统计和当前数据库对不上，“空字典类型是否允许迁移”也还没写成明确规则。

### 5.4 施工单位填报建材产品

相关 change：`materials-project-product`、`materials-record`、`batch-import-dialog`、`system-product`、`system-product-dict`

未解决问题：

1. `[高][前端/后端]` 建材产品列表“重置信息确认状态”按钮及对应后端逻辑尚未实现。
2. `[中][前端/后端/数据库]` 产品“规格”和“单位”现在还没有统一成最终模型，当前只是先从 `master.sys_product.unit` 自动带单位，能用但还不够稳。
3. `[中][前端/后端]` 供应商名称在部分场景应展示“无”的业务口径，仍未完成录入、保存、导出全链路核对。
4. `[低][文档]` `materials-project-product/tasks.md` 尚未回写，仍混有已经完成但未标记的旧任务。
5. `[中][数据库]` 历史数据中仍有少量记录同时缺失 `batch_no` 和 `batch_date`，是否需要治理仍待确认。
6. `[中][前端/后端/文档]` 备案产品导入能力已在 `materials-record` 页面接入 `batch-import-dialog`，但其他复用场景的接入约束与验收口径仍未统一回写。
7. `[中][后端/数据库]` `send_flag` 是否为正式业务字段仍待确认。
8. `[中][后端/数据库]` 企业删除口径尚未全链路统一到 `del_flag + company_type`。
9. `[低][前端/后端/文档]` 批量导入组件本身还没彻底收尾：怎么用、哪些场景能复用、联调结果、边界测试结论都还没写清，报错提示和逐行反馈体验也还有优化空间。
10. `[中][文档]` 产品模块的文档方向现在还是双轨：`system-product` 已经是实际运行方案，但 `system-product-dict` 还保留着“继续按字典表实现”的旧说法，两边没有统一。
11. `[中][前端/文档]` 当前系统里还同时保留 `system/product/index` 和 `system/product/index-dict` 两个产品入口，用户能从两套菜单进入两套页面，最终应该保留哪一套还没定下来。

### 5.5 代理商 / 生产单位审核建材产品

相关 change：`materials-project-product`、`message-center`

未解决问题：

1. `[高][运维]` 运行侧仍需在 SnailJob 后台配置 `matProductTimeoutReminderJobExecutor` 定时任务；在完成该配置前，建材产品“超 48 小时未审核”的超时提醒逻辑虽已落代码，但不会自动触发。

### 5.6 消息提醒与消息中心

相关 change：`message-center`

未解决问题：

1. `[中][后端]` 公告现在还没有“按用户记录已读状态”的能力，所以统一提醒里只能把公告当成只读信息展示，不能像普通消息那样算未读数。
2. `[中][后端/数据库]` 消息历史迁移还没彻底收口，后面还要兼容旧表 `test.t_message` 和少量非纯数字用户标识。
3. `[中][后端]` 消息模板目前只是把表结构先迁过来了，真正按模板发送消息的运行能力还没接上。

### 5.7 通知公告、库存信息、采购信息发布

相关 change：`notice-announcement`

未解决问题：

1. `[中][数据库/文档]` 开发库里的通知公告表现在还是“草稿也会带发布时间、状态默认就是 published”的运行口径；如果后面要改成“草稿没有发布时间”，不能只改文档，得连实现一起改。
2. `[高][数据库]` 开发库 `msg_notice_publish` 尚未补齐主键/索引定义，数据库工程化层面仍有优化空间。
3. `[中][数据库/前端]` 开发库 `msg_publish_status` 字典仍缺少 `draft`，前端目前仅通过本地兜底文案显示草稿状态。

### 5.8 查看备案产品

相关 change：`materials-record`

未解决问题：

1. `[高][数据库]` 备案证号 `record_no` 现在还没有真正加上数据库唯一约束，开发库和 PostgreSQL 源库都还能查到 3 组重复数据，说明唯一索引脚本还没落下去。
2. `[中][后端/数据库]` `send_flag` 是否为正式业务字段仍待确认。
3. `[中][后端/数据库]` 企业删除口径尚未全链路统一到 `del_flag + company_type`。

### 5.9 统计分析

相关 change：`statistics-center`

未解决问题：

1. `[低][前端/后端/文档]` 导出“工作调度表”和“信息确认情况表”当前仍不纳入本次交付，后续需要以独立需求补齐导出范围、模板格式、字段口径和接口实现。
2. `[低][文档]` `purchase_price_analysis` 仍属于超出操作手册范围的增量能力，需继续在 proposal/design/spec 中明确范围说明。

### 5.10 质量追溯与外部数据链路

相关 change：`quality-trace`、`data-screen`

未解决问题：

1. `[中][文档]` 质量追溯相关 proposal、design、spec、tasks 还没有把四个页面当前到底做到什么程度写清楚。
2. `[中][前端/后端]` “缺陷建材使用情况”“缺陷建材厂家”这两个页面目前更像占位页，或者优先依赖外部数据，自己的主库闭环还没形成。
3. `[高][数据库/后端]` `master.t_product_relation` 虽然已经有 1058 条数据，但 `quality_trace_id` 现在全部为空，说明“追溯记录和产品关系真正关联上”这件事还没做成。
4. `[高][数据库/后端]` `master.t_quality_trace` 现在 616 条活动数据里，`batch_no` 和 `product_name` 都是空的，导致页面想按“生产批号”“检验产品名称”查和展示时都不稳定。
5. `[低][文档]` `data-screen` 本轮未发现新增未完成项，可暂不单列遗留问题。

### 5.11 底层主数据迁移与基础支撑

相关 change：`dict-migration-test-to-master`、`system-region`、`kingbase-master-table-sync`、`user-role-migration`

未解决问题：

1. `[中][前端/后端]` 地区搜索语义、根节点口径和组件统一仍待下一轮收口。
2. `[高][数据库]` 金仓开发库 `base_province` 仍存在 4 条活动态 `en_code='110228001'` 重复记录，且未查到 `uk_base_province_en_code_active` 等索引，说明地区清洗和唯一约束脚本尚未真正落库。
3. `[中][文档/数据库]` `kingbase-master-table-sync` 现在已经不是“还有表没迁过来”，而是“表都有了，但文档还停在旧阶段”；另外像 `base_message`、`base_message_receive`、`sys_logininfor` 这种运行中会继续增长的表，也不能再按“必须和源库完全同数”来验收了。
4. `[中][文档/数据库]` 用户迁移和角色映射现在的真实状态是“做了一部分但没完全闭环”，proposal 还没有把这个现实状态写出来。
5. `[中][文档/数据库]` 字典迁移相关文档还没把三件事写实：迁移结果统计、空字典类型怎么验收、编码切换最后怎么闭环。

## 6. 优先级建议

### 6.1 第一优先级：直接影响主链路验收

1. `materials-project` 的施工单位上下文自动带出、工程编码筛选口径统一。
2. `materials-project-product` 的重置信息确认状态、规格单位模型、供应商名称口径收口。
3. `materials-record` 的 `record_no` 唯一索引补齐、`send_flag` / 删除口径收口。
4. `message-center` 的公告已读模型、历史迁移兼容、模板运行时支持。
5. `notice-announcement` 的数据库结构、字典落库与系统公告权限编码统一。

### 6.2 第二优先级：影响提案可信度和后续验收

1. `quality-trace` 的四页真实状态回写与数据口径收口。
2. `user-role-migration` 的文档状态修正、差额核对和 rollback 记录。
3. `dict-migration-test-to-master` 的验收口径回写。
4. `system-product` / `system-product-dict` 的方向统一。

### 6.3 第三优先级：工程化与体验优化

1. `batch-import-dialog` 的使用文档、边界测试和错误提示体验。
2. `system-region` 的搜索语义与组件统一。
3. `statistics-center` 的范围说明整理。

## 7. 使用说明

1. 本文件只保留“当前仍成立”的问题，不重复记录已经被代码或数据库事实证明已完成的旧任务。
2. 若某项问题已在对应 change 的 `issues.md` 修复完成，应同步从本文件删除或改状态。
3. 若后续仍希望同时保留“按 change 查看”和“按业务流程查看”两种视角，可继续以本文件作为总入口、各 change 的 `issues.md` 作为明细来源。
