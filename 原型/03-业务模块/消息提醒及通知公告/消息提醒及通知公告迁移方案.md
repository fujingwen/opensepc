# 消息提醒及通知公告迁移方案

## 1. 模块范围

本模块对应迁移清单中的以下功能：

- 消息提醒及通知公告-审核消息提醒
- 超48小时未审核消息提醒
- 消息提醒及通知公告-通知
- 消息提醒及通知公告-系统公告
- 预警信息

## 2. 当前现状

### 2.1 已有能力

当前新系统已经形成两条真实承载链路：

1. 消息中心
   - 运行表：`master.base_message`、`master.base_message_receive`、`master.base_message_template`
   - 当前记录数：`26 / 185493 / 0`
   - 顶部铃铛已通过 `/message/notification/summary` 汇总真实消息与公告摘要
   - 手工消息与建材业务消息已汇聚到统一提醒入口
2. 通知公告
   - 运行表：`master.msg_notice_publish`
   - 当前活动数据：`4`
   - 当前已经区分：
     - 系统公告：只读查看
     - 库存信息：草稿 -> 发布
     - 采购信息：草稿 -> 发布

### 2.2 仍需兼容或补齐的历史部分

- 历史迁移仍需兼容 `test.t_message` 和少量非纯数字用户标识
- 公告目前没有用户维度已读关系，统一提醒中的公告暂不参与未读统计
- 消息模板表虽然已落库，但运行时模板发送能力尚未接入

### 2.3 当前未收口的能力

- 预警信息页面
- 公告已读关系
- 消息模板运行时发送
- 超48小时未审核提醒的运行侧任务配置

## 3. 当前承载判断

### 3.1 系统公告与业务公告

- 不再建议继续映射到 `master.sys_notice`
- 当前真实方向已是 `notice-announcement`：
  - 系统公告只读
  - 库存/采购公告按业务角色维护
  - 页面与后端都已取消手工选企业

### 3.2 通知与消息中心

- 现状：已承载到 `master`
- 当前真实运行模型：
  - `base_message`：消息主表
  - `base_message_receive`：收件箱/提醒落点
  - `base_message_template`：模板结构预留
- 当前剩余问题不再是“有没有承载表”，而是：
  - 历史迁移闭环
  - 公告已读模型
  - 模板能力接入

### 3.3 审核消息提醒与超48小时提醒

- 审核消息提醒：已实现主干链路，建材审核相关消息已进入统一提醒
- 超48小时提醒：代码已实现，`matProductTimeoutReminderJobExecutor` 已存在
- 当前差异：运行侧仍需在 SnailJob 后台完成任务配置，否则不会自动触发

### 3.4 预警信息

- 现状：仍未形成独立页面或明确承载模型
- 结论：仍应保留为后续独立收口项

## 4. 推荐收尾顺序

### 4.1 第一优先级

- 公告已读关系模型
- `test.t_message` 历史迁移兼容
- `msg_notice_publish` 的字典、索引与表结构工程化补齐

### 4.2 第二优先级

- 消息模板运行时发送能力
- SnailJob 超时提醒任务配置与运行验收

### 4.3 第三优先级

- 预警信息页面与数据来源边界
