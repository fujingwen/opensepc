# 产品管理

## 当前实现结论

产品管理已经基于 `master.sys_product` 落地，不再是“继续沿用字典页”的待设计状态。

当前真实实现：

- 后端接口：`/system/product`
- 主页面：`system/product/index`
- 兼容页面：`system/product/index-dict`
- 查询权限：`system:product:list` 与 `materials:product:list` 共享可访问

## 数据现状

| 项目 | 数值 |

| `master.sys_product` 总记录数 | `38476` |

说明：

- 表内已经形成 `category / product / spec` 三层口径。
- 但历史数据并不完全规范，不能假设所有记录都严格满足标准树结构。

## 真实业务口径

### 左侧类别树

左侧按 `node_type='category'` 查询并组织树形结构，支持类别新增、编辑、删除。

### 右侧产品树

右侧不是简单按“标准树路径”查子节点，而是按以下双通道兼容历史数据：

- `tree_path` 命中当前类别
- `category_id = 当前类别ID`

这是当前实现中的关键兜底逻辑，否则部分历史类别会出现“右侧产品为空”的假象。

### 权限口径

产品查询接口目前采用共享权限：

- 系统管理页可用 `system:product:list`
- 建材产品业务页可用 `materials:product:list`

因此产品管理已经不只是系统管理内部能力，也承担建材填报链路中的基础数据选择能力。

## 页面与接口

### 页面

| 页面 | 说明 |
|------|------|
| `system/product/index` | 当前主页面，基于 `sys_product` |
| `system/product/index-dict` | 保留的旧字典页面，用于兼容/对照 |

### 核心接口

| 接口 | 说明 |
|------|------|
| `GET /system/product/category/list` | 左侧类别树 |
| `GET /system/product/list/{categoryId}` | 右侧产品树 |
| `GET /system/product/lazy/{parentId}` | 懒加载子节点 |
| `POST /system/product/category` | 新增类别 |
| `POST /system/product` | 新增产品/规格 |
| `PUT /system/product` | 编辑 |
| `DELETE /system/product/{id}` | 删除 |

## 需要替换掉的旧说法

以下旧结论已经不再成立：

- “产品管理仍应新建 `sys_product` 表”
- “当前仍主要依赖 `base_dictionarytype/base_dictionarydata`”
- “产品页尚未形成独立实现”

当前正确说法应为：

- `sys_product` 已经存在并已投入使用
- 系统管理产品页已切换到 `sys_product`
- 字典页只保留为兼容页面，不再代表最终方向

## 剩余问题

1. `sys_product` 历史数据仍存在 `tree_path`、层级挂载不规范的问题，当前依赖查询兼容兜底。
2. 数据库中同时存在 `system/product/index` 与 `system/product/index-dict` 两个菜单入口，最终保留哪一个仍需收口，否则用户入口会出现双轨并存。
