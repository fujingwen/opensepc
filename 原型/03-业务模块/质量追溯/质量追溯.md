# 质量追溯模块分析

## 1. 模块概述

质量追溯模块当前已经有正式的 OpenSpec 方案、数据库承载表和后端查询实现，包含四个页面：

1. 抽测缺陷建材产品
2. 检测缺陷建材产品
3. 缺陷建材使用情况
4. 缺陷建材厂家

当前不应再按“尚未实现”或“仅旧库分析”来描述，而应按“已落地但数据闭环未完成”理解。

## 2. 当前数据落点

### 2.1 核心表

| 表名 | 说明 | 当前记录数 |
| --- | --- | ---: |
| `master.t_quality_trace` | 质量追溯主表 | 616 |
| `master.t_product_relation` | 追溯关联表 | 1058 |

### 2.2 关键事实

1. `master.t_quality_trace` 已存在，不再是“待迁移到主库”的状态。
2. `master.t_product_relation` 也已存在，不再是“仅旧库存在”。
3. 但当前关系仍未闭环：
   - `quality_trace_id` 为空 `1058`
   - `legacy_check_product_id` 非空 `1058`
4. `master.t_quality_trace` 当前数据质量仍有明显缺口：
   - `batch_no` 为空 `616`
   - `product_name` 为空 `616`

## 3. 页面口径

### 3.1 抽测缺陷建材产品

- 当前支持：查询、导入、删除
- 核心字段：工程名称、检测项目、检验产品名称、生产厂家、生产批号、检测日期、报告日期、检验参数、结论、有无对比数据
- 当前风险：`product_name`、`batch_no` 普遍为空，会影响列表展示和筛选可用性

### 3.2 检测缺陷建材产品

- 当前支持：查询、删除、复检合格
- 与抽测页共用主表 `master.t_quality_trace`
- 当前风险与抽测页一致，仍受主表字段缺失影响

### 3.3 缺陷建材使用情况

- 当前页面已按“质量追溯记录 + 关联产品 + 工程项目信息”方向设计
- 但当前 `t_product_relation` 仍主要依赖 `legacy_check_product_id` 做弱关联，尚未形成稳定的一对一闭环

### 3.4 缺陷建材厂家

- 当前页面已纳入模块范围
- 但生产厂家相关展示仍依赖主表和关联关系的当前数据质量

## 4. 当前关系判断

当前真实口径更接近下面这条链路：

```text
t_quality_trace
  └─ 通过 t_product_relation 的 legacy_check_product_id / quality_trace_id 弱关联
       └─ 找到 t_project_product
            └─ 再关联 t_project、t_companyinfo
```

注意：

1. 这里不是一条已经被数据库完全证明的稳定强关系。
2. 当前“有无对比数据”是代码层按弱关联存在性判断出来的。
3. 若后续要做真正的复检闭环或严谨统计，仍需先把 `quality_trace_id` 回填规则收口。

## 5. 当前结论

1. 质量追溯不是“未实现”，而是“页面、表、接口已存在，但数据闭环和文档未收尾”。
2. 当前最核心的问题不是建不建表，而是：
   - `t_quality_trace` 关键字段缺值
   - `t_product_relation` 仍是弱关联
   - 四个页面的真实实现状态没有完全回写到文档
