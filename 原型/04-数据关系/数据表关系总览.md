# 数据表关系总览

## 当前基线

本文只保留当前收尾阶段仍然有效的关系，不再复述已被推翻的历史假设。

截至 2026-04-10，建议把数据关系理解为以下几条主链路。

## 1. 建材填报主链路

```text
t_project
  └─ 1:N ──> t_project_product
                └─ N:1 ──> t_record_product（通过 record_no 业务关联，当前仍需唯一约束补强）
```

说明：

- `t_project` 已是工程项目主表。
- `t_project_product` 已是建材产品主表。
- `t_record_product` 是备案产品承载表，但 `record_no` 仍存在活动态重复。

## 2. 质量追溯链路

```text
t_quality_trace
  └─ 弱关联 ──> t_product_relation
                    └─ product_id / F_ProductId ──> t_project_product
```

说明：

- `t_product_relation` 已存在且有数据。
- 但当前 `quality_trace_id` 基本未闭环，更多依赖历史字段做弱关联。
- 因此不能把这条链路描述成“已经完全可靠的一对一追溯映射”。

## 3. 消息中心链路

```text
base_message
  └─ 1:N ──> base_message_receive
```

说明：

- 这是当前铃铛、收件、提醒的主链路。
- `base_message_template` 结构已存在，但目前仍为空表。

## 4. 通知公告链路

```text
msg_notice_publish
```

说明：

- 通知公告不再以 `sys_notice` 为主口径。
- 公告发布、库存信息、采购信息都应优先围绕 `msg_notice_publish` 理解。

## 5. 系统基础数据链路

```text
sys_user / sys_role / sys_dept / sys_menu / sys_dict_*
base_province
sys_product
```

说明：

- 行政区划当前使用 `base_province`
- 产品管理当前使用 `sys_product`

## 当前仍需特别标记的弱点

1. `base_province.en_code` 活动态重复与索引缺失仍未治理完成。
2. `sys_product` 历史层级结构不规范，查询依赖兼容兜底。
3. `t_record_product.record_no` 缺少可靠活动态唯一约束。
4. `t_product_relation -> t_quality_trace` 仍未形成稳定闭环。
5. `msg_notice_publish` 结构工程化仍未完全收口。
