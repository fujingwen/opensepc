# test 与 master 迁移盘点

## 当前结论

这份盘点不应再按“哪些表在 master 中缺失”来写，而应改成“哪些模块已经承载到位，哪些还存在数据质量或闭环问题”。

截至 2026-04-10，主库承载情况已经明显前移：

| 模块 | 当前主库承载 |
|------|--------------|
| 用户/组织/角色/菜单/字典 | `sys_user / sys_dept / sys_role / sys_menu / sys_dict_*` |
| 行政区划 | `master.base_province` |
| 产品管理 | `master.sys_product` |
| 企业信息 | `master.t_companyinfo` |
| 工程项目 | `master.t_project` |
| 建材产品 | `master.t_project_product` |
| 备案产品 | `master.t_record_product` |
| 质量追溯 | `master.t_quality_trace` |
| 追溯关联 | `master.t_product_relation` |
| 消息中心 | `master.base_message / base_message_receive / base_message_template` |
| 通知公告 | `master.msg_notice_publish` |

## 已不成立的旧判断

以下历史结论已经过期：

- `master` 中不存在 `t_project`
- `master` 中不存在 `t_quality_trace`
- 消息中心在 `master` 中没有承载表
- 通知公告仍需研究是否映射到 `sys_notice`
- `kingbase-master-table-sync` 仍处于“缺表待补”阶段

## 当前更准确的迁移状态

### 已基本完成主库承载

- `t_project`
- `t_project_product`
- `t_record_product`
- `t_quality_trace`
- `t_product_relation`
- `base_message`
- `base_message_receive`
- `sys_dict_type`
- `sys_logininfor`

### 当前问题主要变成“数据质量/约束/闭环”

典型问题包括：

- `t_record_product.record_no` 仍有重复活动数据，缺少活动态唯一约束
- `t_quality_trace` 与 `t_product_relation` 仍未形成稳定闭环
- `msg_notice_publish` 表结构工程化不足
- `base_province` 仍存在重复 `en_code`
- `sys_product` 历史层级结构不规范

### 动态表不再以“行数完全一致”作为验收标准

如 `base_message`、`base_message_receive`、`sys_logininfor` 这类运行中持续变化的表，当前更合理的验收方式应为：

- 结构一致
- 索引/约束一致
- 当前基线已记录

而不是要求与源库始终保持同一时刻完全同数。

## 仍需继续验证的点

1. `t_product_relation.quality_trace_id` 当前全为空，追溯关系仍是弱关联。
2. `t_quality_trace.batch_no`、`product_name` 仍大量为空，页面可展示但闭环不足。
3. 用户、角色、菜单等迁移后，仍需继续核对 `original_id` 与授权链路是否完整。
