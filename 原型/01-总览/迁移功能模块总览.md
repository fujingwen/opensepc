# 迁移功能模块总览

## 1. 依据

- 功能清单来源：`openspec/原型/02-来源资料/建材平台系统适配迁移功能清单-3.20.docx`
- 代码范围：
  - 后端：`construction-material-backend/hny-modules/hny-base`、`hny-materials`、`hny-system`、`hny-admin`
  - 前端：`construction-material-web/src/views`、`construction-material-web/src/api`
- 数据库：`building_supplies_supervision`
  - 新系统 schema：`master`
  - 旧系统 schema：`test`

## 2. 核心结论

### 2.1 已有较完整承载的模块

- 基础数据中的生产企业、代理商、施工企业已经有前后端实现，核心承载表为 `master.t_companyinfo`
- 组织、部门、岗位、角色、账号、登录已经由 `sys_user`、`sys_role`、`sys_dept`、`sys_post`、`sys_menu` 承载
- 行政区划、产品管理、通知公告已经在 `master` 中有真实表和菜单
- 备案产品管理已经基本落到 `master.t_record_product`

### 2.2 已开发但方向需要纠偏的模块

- 工程项目模块当前代码已统一围绕 `t_project`，主链路以 `test.t_project -> master.t_project` 为准
- 施工企业填报建材数据当前代码已统一围绕 `master.t_project_product`，但业务口径仍需继续收口
- 质量追溯、消息中心、通知公告、统计分析、数据大屏均已有代码与数据承载，但文档、数据质量或链路验收仍未完全收尾

### 2.3 尚未真正承接的模块

- 预警信息
- 复检闭环
- 数据共享
- 工程造价信息查询

## 5. 下一步迁移方案

### 5.1 第一阶段：纠偏现有已开发模块

1. 工程项目、建材填报、通知公告、消息中心等主链路文档继续按真实实现回写
2. 先补齐 `t_record_product` 唯一索引与重复数据
3. 先补齐 `base_province` 唯一约束和脏数据清理
4. 继续收口 `t_project_product`、`t_quality_trace`、`t_product_relation` 的业务口径

### 5.2 第二阶段：补齐旧库核心业务表

1. 持续校验 `master.t_project` 数据完整性与业务字段口径
2. 校验 `master.t_quality_trace`、`master.t_product_relation` 的弱关联口径
3. 继续补写 `message-center` 与 `kingbase-master-table-sync` 的收尾验收结论
4. 核定 `t_product_relation` 是继续保留弱关联，还是补充稳定映射规则

### 5.3 第三阶段：补未承接功能

1. 预警信息
2. 复检闭环
3. 数据共享
4. 工程造价查询
