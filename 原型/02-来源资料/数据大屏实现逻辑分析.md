# 数据大屏实现逻辑分析

## 1. 文档目的

本文说明当前仓库内“建材数据大屏”的真实实现链路，包括：

- 菜单如何打开大屏页面
- 前端页面如何加载、组装和渲染数据
- 后端接口如何拆分统计能力
- 当前新大屏与旧版 test 模式 DataV 大屏的对应关系

本文基于以下代码与资料整理：

- `construction-material-web/src/views/dashboard/materials/index.vue`
- `construction-material-web/src/api/dashboard/materials.js`
- `construction-material-web/src/utils/menuTarget.js`
- `construction-material-backend/hny-modules/hny-materials/src/main/java/com/hny/materials/controller/MatDashboardController.java`
- `construction-material-backend/hny-modules/hny-materials/src/main/java/com/hny/materials/service/impl/MatDashboardServiceImpl.java`
- `construction-material-backend/hny-modules/hny-materials/src/main/resources/mapper/MatDashboardMapper.xml`
- `test.blade_visual_*` 中的旧版 DataV 大屏配置

## 2. 整体实现结构

当前大屏不是旧版 DataV 配置式大屏，而是“前端固定布局 + 后端拆分接口”的实现。

整体链路如下：

1. 用户从菜单点击“大屏”
2. 菜单逻辑将其打开到独立路由 `/screen/dashboard`
3. 路由加载前端页面 `views/dashboard/materials/index.vue`
4. 页面挂载后并行调用：
   - `GET /materials/dashboard/summary`
   - `GET /materials/dashboard/rates`
   - `GET /materials/dashboard/regions`
   - `GET /materials/dashboard/types`
   - `GET /materials/dashboard/lists`
5. 前端拿到数据后分别渲染卡片、环图、地图、榜单和趋势图

这说明当前实现的核心思想是：

- 前端按模块拆分请求
- 后端按模块提供只读统计接口
- 页面支持模块级失败降级

## 3. 页面入口与打开方式

### 3.1 路由入口

大屏独立承载在：

- `/screen/dashboard`

对应页面：

- `construction-material-web/src/views/dashboard/materials/index.vue`

### 3.2 菜单打开逻辑

当前菜单辅助逻辑会把以下菜单路径统一映射到独立页：

- `/dashboard`
- `/dashboard/materials`
- `/dashboard/materials/index`

最终实际打开：

- `/screen/dashboard`

因此现在从菜单点击大屏，走的是独立页面，而不是在当前业务页签内普通跳转。

## 4. 前端实现逻辑

### 4.1 页面结构

前端页面采用三列网格布局：

- 左列：平台汇总概览、区市统计、产品类别统计
- 中列：审核核心指标、青岛地图
- 右列：最近项目审核、缺陷建材榜单、月度趋势

### 4.2 生命周期

页面挂载时：

1. 调用 `loadOverview()`
2. 通过 `Promise.allSettled` 并行请求 5 个接口
3. 将返回值分别写入：
   - `summary`
   - `confirmationRate`
   - `approvalRate`
   - `regionStats`
   - `typeStats`
   - `monthlyTypeTrend`
   - `projectAuditList`
   - `defectProductList`
4. `nextTick()` 后初始化图表和地图
5. 注册 `resize` 监听，窗口变化时统一重绘

页面卸载时会销毁 ECharts 和 L7 场景。

### 4.3 图表与地图

- 汇总卡片使用文本卡片渲染
- 审核指标使用 ECharts 环形图
- 区市统计使用饼图
- 产品类别统计使用柱状图
- 月度趋势使用多折线图
- 地图使用 `@antv/l7`、`@antv/l7-maps` 与本地 `qingdao.json`
- 地图支持 3D 挤出、悬浮弹窗、边界线、标签偏移
- 右侧两个榜单支持自动向上滚动、悬浮暂停、移出恢复

## 5. 后端实现逻辑

## 5.1 接口入口

当前大屏只保留 5 个接口：

- `GET /materials/dashboard/summary`
- `GET /materials/dashboard/rates`
- `GET /materials/dashboard/regions`
- `GET /materials/dashboard/types`
- `GET /materials/dashboard/lists`

控制器：

- `MatDashboardController`

权限控制：

- `materials:dashboard:view`

## 5.2 服务层逻辑

服务层按模块拆分：

- `getSummary()`
- `getRates()`
- `getRegionStats()`
- `getTypes()`
- `getLists()`

特点是：

- 无分页
- 无筛选条件
- 无离线汇总表
- 按展示模块直接聚合

## 5.3 Mapper 统计口径

### 1. 汇总卡片 `selectSummary`

数据来源：

- `master.t_project`
- `master.t_project_product`
- `master.t_record_product`

### 2. 信息确认率 `selectConfirmationRate`

- 分母：`master.t_project_product` 有效记录总数
- 分子：`record_no` 非空记录数

### 3. 审核通过率 `selectApprovalRate`

- 已确认记录中
- `check_status = 1`
- 且 `supplier_check_status = 1` 或为空

### 4. 区市统计 `selectRegionStats`

当前采用固定青岛 10 区市字典，并基于 `project_address` 文本匹配归属区市。

为避免超时，当前 SQL 已调整为：

- 先对 `t_project_product` 按项目聚合
- 再关联项目归属区市
- 最后按区市汇总

### 5. 产品类别统计 `selectTypeStats`

通过：

- `t_project_product.product_type`
- 关联 `base_dictionarytype`

统计产品类别数量，取前 10。

### 6. 月度趋势 `selectMonthlyTypeTrend`

逻辑分三步：

1. 生成近 12 个月月份序列
2. 统计近 12 个月出现次数最多的前 8 个产品类别
3. 对每个月和每个类别做补齐，没有值则补 0

### 7. 最近项目审核榜单 `selectProjectAuditList`

来源：

- `master.t_project_product`
- 左联 `master.t_project`

### 8. 缺陷建材榜单 `selectDefectProductList`

来源：

- `master.t_quality_trace`

筛选条件：

- `conclusion_mark = 'n'`

## 6. 与旧版 test 大屏的关系

旧版 test 模式大屏是 DataV 配置式大屏，特点是：

- 页面配置存放在 `test.blade_visual`、`test.blade_visual_config`
- 组件逐个配置位置、样式、接口地址
- 每个图表或卡片单独请求自己的 `/api/screen/*` 接口

而当前新实现是：

- 前端布局固定
- 前端仍按模块拆分请求
- 但由当前项目的业务代码直接维护
- 不再使用 DataV 配置表驱动

## 7. 结论

当前数据大屏已经形成一条清晰、可维护的实现链路：

- 菜单独立打开
- 单页渲染
- 拆分接口并行加载
- 前端固定布局
- 后端按模块聚合

与旧版 test 大屏相比，当前新大屏延续了业务分区思路，但实现方式已经明确切换为代码式维护，不再保留历史综合接口结构。
