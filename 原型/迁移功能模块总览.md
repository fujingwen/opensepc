# 迁移功能模块总览

## 1. 依据

- 功能清单来源：`openspec/原型/原型文件/【建材平台】系统适配迁移功能清单-3.20docx.docx`
- 代码范围：
  - 后端：`construction-material-backend/hny-modules/hny-base`、`hny-materials`、`hny-system`、`hny-admin`
  - 前端：`construction-material-web/src/views`、`construction-material-web/src/api`
- 数据库：`building_supplies_supervision`
  - 新系统 schema：`master`
  - 旧系统 schema：`test`

## 2. 核心结论

### 2.1 已有较完整承载的模块

- 基础数据中的生产企业、代理商、施工企业已经有前后端实现，核心承载表为 `master.t_companyinfo`
- 组织、部门、岗位、角色、账号、登录已经由 `sys_user`、`sys_role`、`sys_dept`、`sys_post`、`sys_menu` 承载
- 行政区划、产品管理、通知公告已经在 `master` 中有真实表和菜单
- 备案产品管理已经基本落到 `master.t_record_product`

### 2.2 已开发但方向需要纠偏的模块

- 工程项目模块当前代码围绕 `mat_project`，但迁移目标应回到 `test.t_project -> master.t_project`
- 施工企业填报建材数据当前代码围绕 `mat_product`，但迁移目标应回到 `test.t_project_product -> master.t_project_product`
- 代理商、生产企业审核接口只有预留，没有真实审核流

### 2.3 尚未真正承接的模块

- 质量追溯
- 预警信息
- 消息中心
- 统计分析
- 数据大屏
- 复检闭环
- 数据共享
- 工程造价信息查询

## 5. 下一步迁移方案

### 5.1 第一阶段：纠偏现有已开发模块

1. 工程项目模块统一回调到 `t_project`
2. 建材填报模块统一回调到 `t_project_product`
3. 停止继续扩展 `mat_project`、`mat_record`、`mat_product` 方向
4. 先补齐 `t_record_product` 差异数据

### 5.2 第二阶段：补齐旧库核心业务表

1. 建立并迁移 `master.t_project`
2. 建立并迁移 `master.t_quality_trace`
3. 评估 `test.t_message`、`test.base_message`、`test.base_messagereceive` 在 `master` 的最终承载策略
4. 核定 `t_product_relation` 是保留旧结构还是重建关系规则

### 5.3 第三阶段：补未承接功能

1. 审核流与消息提醒
2. 预警信息
3. 质量追溯四个子模块
4. 统计分析与大屏
5. 数据共享与工程造价查询

## 6. 建议拆分的实施任务

### 6.1 代码纠偏任务

- `materials/project` 从 `mat_project` 改为 `t_project`

### 6.2 数据迁移任务
